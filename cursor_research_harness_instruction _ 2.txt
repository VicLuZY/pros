CURSOR BUILD INSTRUCTION
Research and Deliverable Harness for Research-Grade Knowledge Work

You are Cursor. Build a production-ready scaffold and harness for a knowledge-work agent that behaves like a research-grade assistant, not a single-pass chatbot. The goal is not to imitate any proprietary hidden chain-of-thought. The goal is to reproduce the external behavior that makes high-end research systems feel strong: deliberate planning, multi-step web research, source tracking, evidence grading, applicability assessment, interdisciplinary collaboration, iterative refinement, domain-specific reasoning, and high-fidelity deliverable generation.

This harness must not stop at “good reasoning.” It must reliably transform research into the exact kind of output the user asked for. If the user asks for a proposal, it should produce a strong proposal. If the user asks for an engineering report, it should produce an engineering report. If the user asks for an accounting spreadsheet, it should produce a structured spreadsheet-ready output with formulas, assumptions, reconciliation logic, controls, and review notes. If the user asks for a cross-domain deliverable, the harness must assemble and coordinate multiple specialists.

The standard of success is not merely “the model thought deeply.” The standard of success is:
- the harness gathered the right domain knowledge
- the harness used the right evidence method for the task
- the harness used the right expert lenses
- the harness matched the requested output type and audience
- the harness produced a deliverable that is useful, credible, well-structured, and tailored to the user’s actual request
- the harness can explain which sources it searched, how many it searched, and why the final output should be trusted

Non-negotiable outcomes:
- The system must search the internet when the task requires fresh, niche, or current information.
- The system must report how many sources it searched.
- The system must cite the sources it actually used for factual claims.
- The system must iterate until it has either satisfied the task or can clearly explain what blocked completion.
- The system must keep a concise visible reasoning summary and action log, but must not expose hidden chain-of-thought.
- The system must generate an output package that conforms to the user’s requested deliverable type.
- The system must support cross-domain collaboration between multiple expert agents when needed.
- The system must distinguish between research completeness and deliverable readiness. A task is not done just because research is done.
- The system must make evidence quality and applicability explicit, not implicit.
- The system must adapt its writing to the audience, not merely summarize sources.

Research-informed methodology to embed:
- Use a PRISMA-like mindset for transparency of search, screening, inclusion, and versioning.
- Use scoping-review style breadth when the task is exploratory and mapping-oriented.
- Use rapid-review style shortcuts only when time or cost constraints justify them, and make those shortcuts explicit.
- Use living-review style version tracking when the question is likely to change quickly or requires repeat refreshes.
- Use GRADE-like logic for quantitative or claim-heavy evidence, rating confidence through dimensions such as directness, consistency, precision, risk of bias, and relevance.
- Use CERQual-like logic for qualitative or interpretive findings, rating confidence through methodological limitations, coherence, adequacy, and relevance.
- Use evidence-based management logic for business or consulting tasks, explicitly balancing rigor and contextual relevance rather than pretending that a statistically “strong” source is always decision-useful.
- Use team-science logic for cross-domain tasks, including shared goals, explicit role boundaries, a common glossary, conflict handling, trust-building behavior, and a final reconciliation step.
- Use technical-writing and policy-brief logic to ensure the final deliverable is audience-aware, concise where needed, decision-led, and not cluttered by unnecessary background.
- Use spreadsheet-engineering logic so spreadsheet outputs are auditable, structured, versioned, tested, and not dependent on hidden hard-coded values.

Core design philosophy:
- Prefer simple, composable patterns over an overengineered framework.
- Use an orchestrator plus specialists rather than one giant prompt.
- Separate research from deliverable generation, but connect them with a strict evidence and requirements contract.
- Treat output fidelity as a first-class concern, not an afterthought.
- Treat evidence quality, recency, and applicability as first-class concerns, not afterthoughts.
- Use explicit completion criteria, citation rules, verification rules, formatting rules, and stop conditions.
- Persist state so a long task can resume cleanly.
- Optimize for correctness first, then usefulness, then cost and latency.
- Keep the codebase model-provider agnostic.
- Let specialists work in parallel, but require distilled handoff artifacts rather than raw context dumping.
- Force every final deliverable to answer the user’s real ask, not just summarize the research.
- Force every output to clearly separate sourced findings, synthesis, assumptions, calculations, and recommendations.

High-level behavior:
The harness must operate as a stateful workflow with two intertwined tracks:
1. Research track
2. Deliverable track

The research track gathers, grades, and validates knowledge.
The deliverable track determines what the user actually wants produced, what “good” looks like for that artifact, and how multiple specialists should contribute.

The system must not assume that one universal final-answer template is sufficient. It must derive an output contract from the task. The harness should be capable of producing, at minimum:
- research reports
- proposals
- engineering memos
- engineering reports
- technical comparisons
- accounting or finance analysis tables
- spreadsheet-ready workbooks or workbook specs
- business cases
- client emails or executive summaries
- policy briefs or decision briefs
- multi-section deliverables where different parts require different expert lenses

RESEARCH MODE SELECTION
Every run must explicitly choose one research mode before major search work begins:
- quick-answer mode: for narrow factual lookups with limited synthesis
- scoped-research mode: for exploratory mapping, landscape scans, and unfamiliar domains
- rapid-evidence-synthesis mode: for decision-oriented research on a short timeline
- deep-synthesis mode: for higher-rigor, multi-source, high-stakes questions
- living-research mode: for topics expected to change frequently or require repeat refresh

The selected mode must control:
- search breadth
- screening rigor
- evidence table depth
- reviewer intensity
- how confidence is reported
- how change tracking works between versions

STAGE 0. TASK INTAKE
- Parse the user task.
- Extract objectives, constraints, desired output format, freshness needs, domains, audience, risk level, tone, granularity, and any explicit acceptance criteria.
- Create a task record with a unique run ID.
- Decide whether the task requires web research, file input, or both.
- Decide whether the user wants a direct answer, a document, a spreadsheet, a presentation spec, a memo, a proposal, or a mixed artifact.
- Choose the research mode.
- Produce an explicit definition of done.
- Produce a must-answer question list.
- Produce an output contract describing:
  - deliverable type
  - intended audience
  - required sections or tabs
  - required level of domain depth
  - citation expectations
  - whether formulas, tables, figures, assumptions, appendices, or executive summaries are required
  - whether the artifact should optimize for decision speed, technical completeness, compliance defensibility, or operational usability
- Produce an evidence protocol describing:
  - inclusion criteria
  - exclusion criteria
  - preferred source hierarchy
  - freshness threshold
  - evidence-grading approach
  - applicability or transferability rules

STAGE 1. DOMAIN ROUTING AND EXPERT ASSEMBLY
- Detect which expertises are needed.
- Assemble an expert panel from reusable domain agents.
- Support both single-domain and cross-domain work.
- Examples of expert panels:
  - engineering + code compliance + utility coordination
  - accounting + spreadsheet modeling + tax or policy lookup
  - proposal writing + domain specialist + commercial reviewer
  - business strategy + technical SME + executive communications editor
  - technical research + policy brief writer + visual summarizer
- Each specialist must receive a sharply scoped brief.
- Each specialist must return a distilled memo, not a raw chain-of-thought.
- The orchestrator must reconcile specialist outputs into one coherent task plan.
- For cross-domain tasks, create:
  - a shared goals statement
  - a shared glossary for potentially ambiguous terms
  - an ownership map showing which specialist owns which questions
  - an assumptions register
  - a disagreement protocol for handling conflicts between specialists

STAGE 2. PLANNING
- Create a research plan with:
  - key questions
  - likely source categories
  - search strategy
  - fallback strategy
  - verification strategy
  - evidence-grading strategy
  - applicability strategy
  - output-generation strategy
  - collaboration plan among specialists
  - stop conditions
- Break the work into focused sub-questions.
- Produce an evidence coverage matrix keyed to the must-answer questions.
- Produce a deliverable coverage matrix keyed to the output contract.
- Explicitly list which expert owns which section or concern.
- Produce a review protocol that states:
  - what will be screened in or out
  - what kinds of sources are considered primary, secondary, expert commentary, or operational guidance
  - how contradictory evidence will be handled
  - how quantitative and qualitative evidence will be combined

STAGE 3. SEARCH AND GATHER
- Issue search queries.
- Record every query.
- Collect candidate results.
- Normalize URLs.
- Deduplicate by canonical URL and host.
- Fetch pages.
- Extract text content.
- Save metadata:
  - title
  - url
  - domain
  - author if available
  - published or updated date if available
  - retrieval timestamp
  - query that found it
  - status code
  - content hash
  - source type classification
- Maintain these counters:
  - search_queries_issued
  - candidate_results_seen
  - unique_sources_opened
  - unique_domains_opened
  - sources_screened_out
  - sources_in_evidence_ledger
  - sources_cited_in_final
The final answer must always include at least unique_sources_opened and sources_cited_in_final.
- Maintain a PRISMA-like source flow log with counts for identified, opened, excluded, retained, and cited sources.

STAGE 4. SOURCE TRIAGE AND SCREENING
- Score sources for relevance, authority, recency, uniqueness, and domain suitability.
- Prefer primary and official sources when available.
- Flag weak, duplicate, stale, or low-authority sources.
- Identify conflicts between sources.
- Record screening decisions and the reason for exclusion when a candidate is rejected.
- Promote sources into an evidence ledger only if they survive basic quality checks.
- Tag sources by domain so the right specialists consume them.
- For exploratory tasks, allow broader inclusion but label lower-confidence or indirect sources explicitly.
- For decision-heavy tasks, require a stricter source hierarchy and stronger applicability checks.

STAGE 5. REASONED EXTRACTION AND EVIDENCE GRADING
- Extract atomic claims, facts, dates, numbers, and quotations only as needed.
- Store evidence snippets with source references.
- Build a claim-to-evidence map.
- Build a requirement-to-evidence map.
- Build a finding-to-confidence map.
- For each important claim, track:
  - claim text
  - confidence
  - supporting sources
  - contradiction sources
  - freshness sensitivity
  - directness or indirectness
  - applicability notes
  - owning specialist
- Never let the final answer rely on unsupported claims.
- For quantitative or fact-heavy claims, compute a GRADE-like confidence descriptor using dimensions such as:
  - risk of bias
  - directness
  - consistency
  - precision
  - relevance to the user’s context
- For qualitative or interpretive findings, compute a CERQual-like confidence descriptor using dimensions such as:
  - methodological limitations
  - coherence
  - adequacy
  - relevance
- For business or management recommendations, explicitly rate evidence by:
  - rigor
  - contextual relevance
  - operational usefulness
  - consensus level
- For each major recommendation, record:
  - recommendation
  - evidence basis
  - assumptions
  - tradeoffs
  - applicability limits
  - decision impact if wrong

STAGE 6. DELIVERABLE DESIGN
- Convert the output contract into an artifact plan.
- Decide the best structure for the requested deliverable.
- Define sections, tables, tabs, formulas, appendices, and formatting rules.
- If the user request is under-specified, infer the minimum professional structure that best fits the artifact type while staying close to the user’s wording.
- Ensure the structure reflects the audience.
- Distinguish between executive summary and abstract when relevant.
- Ensure the lead section answers “so what?” and not merely “what did we read?”
- Examples:
  - proposal: cover note, understanding, scope, assumptions, exclusions if needed, fee or optional placeholders, differentiators, next steps
  - engineering report: purpose, background, observations, analysis, code or standard references, options, recommendations, limitations
  - policy brief or decision memo: issue, why it matters, findings, options, implications, recommendation, implementation or risk notes
  - spreadsheet: overview tab, assumptions tab, inputs tab, calculations tab, outputs tab, checks tab, version notes, formulas, units, date stamps
  - executive memo: issue, findings, implications, recommendation, concise support
- Create a deliverable blueprint before drafting.
- The blueprint must state which sections are for executives, which are for technical readers, and which are appendices or audit support.

ARTIFACT-SPECIFIC QUALITY RULES
Proposal outputs must:
- demonstrate understanding of the client problem
- show a plausible approach without sounding bloated or generic
- include domain-specific value, not template filler
- distinguish assumptions, scope, and optional extras cleanly
- include commercial and delivery realism

Engineering reports must:
- be logically structured and technically defensible
- define symbols, units, and assumptions clearly
- keep units consistent in tables and graphs
- ensure figures and tables are explicitly referenced in the text
- distinguish observed facts, calculations, engineering judgment, and recommendations
- include a concise abstract or summary when appropriate

Policy briefs and executive memos must:
- target informed non-specialists when appropriate
- use plain language while preserving correctness
- focus on one issue per brief unless the user explicitly requests a broader landscape
- connect findings to options, implications, feasibility, costs, and recommendations
- make the main point visible in the first screen or first half-page

Spreadsheet and accounting outputs must:
- have a clear flow of inputs, processes, and outputs
- enter each input once wherever feasible
- avoid hard-coded constants inside formulas
- keep formulas consistent across ranges
- stage complex calculations through understandable intermediate steps
- include automatic checks and reconciliation logic
- include versioning, review notes, and protection assumptions
- support peer review and downstream auditability

STAGE 7. DRAFT GENERATION
- Generate the deliverable using the evidence ledger and output contract.
- The draft must match the requested artifact, not default back to a generic essay.
- The draft must reflect audience and domain norms.
- The draft must use specialist contributions where appropriate.
- The draft must avoid unsupported filler.
- The draft must clearly distinguish sourced facts from judgment, synthesis, assumptions, or recommendations.
- When helpful, produce both:
  - a polished human-facing artifact
  - a structured machine-facing representation for downstream rendering
- The drafting layer must be able to create:
  - decision-led openings
  - audience-appropriate summaries
  - evidence tables
  - assumptions tables
  - recommendation matrices
  - spreadsheet specifications with formulas and checks

STAGE 8. VERIFICATION
- Run a verifier pass that checks:
  - every major claim has evidence
  - time-sensitive claims are fresh enough
  - conflicting claims are explicitly resolved or surfaced
  - math and derived calculations are reproducible
  - citations point to the correct source entries
  - the artifact actually satisfies the requested output type
  - the structure matches the output contract
  - cross-domain dependencies are not internally inconsistent
  - major recommendations are supported and bounded by assumptions
  - tables, charts, figures, and spreadsheet outputs match the narrative text
  - executive summaries and abstracts are aligned with the body and do not overclaim
- If verification fails, send the task back to search, design, or synthesis as needed.

STAGE 9. QUALITY REVIEW AND ITERATION
- Run one or more reviewer agents depending on the artifact type.
- Separate factual review from craft review.
- Reviewer examples:
  - factual verifier
  - domain SME reviewer
  - output-quality reviewer
  - executive editor
  - spreadsheet auditor
  - commercial reviewer
  - applicability reviewer
  - cross-domain reconciliation reviewer
- Review questions should include:
  - Did we actually answer the user’s request?
  - Did we choose the right research mode?
  - Is the output high-quality for this artifact type?
  - Is the tone right for the audience?
  - Are there weak, vague, or generic sections?
  - Are there unsupported recommendations?
  - Are cross-domain sections coherent with each other?
  - Does the spreadsheet logic reconcile?
  - Are the confidence levels and limitations made clear enough?
  - Are recommendations realistic for the user’s context?
- If reviewers identify substantive gaps, loop back before finalizing.

STAGE 10. SYNTHESIS AND FINAL PACKAGING
- Produce the final artifact in clear, useful form.
- Include citations inline or in a references section when appropriate.
- Include a short method note.
- Include research statistics.
- Include uncertainty notes where evidence is incomplete or conflicting.
- Include artifact metadata describing:
  - output type
  - research mode used
  - expert agents used
  - whether the result is complete, partial, or blocked
  - evidence confidence posture
- If the task requested a file-oriented artifact, emit the structured content needed to render it or generate the file if that path is implemented.
- Include a short “decision confidence” summary for recommendation-heavy outputs.

STAGE 11. PERSIST AND RESUME
- Save the plan, evidence protocol, evidence ledger, source registry, deliverable blueprint, partial draft, reviewer feedback, stats, and next recommended actions.
- Save a change log between iterations so living or repeated runs can report what changed.
- Allow the next run to resume from saved state instead of restarting.

Required agents or modules:
1. Orchestrator
   - owns the run
   - manages stage transitions
   - decides whether to search more, verify, redesign, synthesize, review, or stop
2. Task Interpreter
   - turns the user request into a definition of done and output contract
3. Method Selector
   - selects the research mode and evidence method
4. Domain Router
   - selects which expert agents are needed
5. Planner
   - turns task into plan and sub-questions
6. Researcher
   - issues searches, opens pages, extracts content
7. Ranker or Triage
   - scores and filters sources
8. Evidence Grader
   - scores confidence, directness, relevance, and applicability
9. Specialist Agents
   - reusable experts for fields such as engineering, accounting, finance, policy, proposal writing, spreadsheet modeling, business writing, and technical editing
10. Deliverable Architect
   - maps the request to the right artifact structure and section plan
11. Synthesizer
   - writes the answer or artifact from the evidence ledger and blueprint only
12. Verifier
   - checks evidence coverage, freshness, contradictions, calculations, and citation integrity
13. Quality Reviewer or Critic
   - performs evaluator-optimizer style critique on the draft and can request another research or drafting pass
14. Spreadsheet Auditor
   - validates workbook logic, formulas, unit consistency, assumptions, and reconciliation logic when spreadsheet outputs are involved
15. Style and Audience Editor
   - ensures the final language matches the requested audience and tone without distorting substance
16. Reconciliation Reviewer
   - resolves cross-domain conflicts and ensures the final story is coherent
17. State Manager
   - persists and restores runs, checkpoints, logs, counters, drafts, and review artifacts

Minimal stopping logic:
Do not stop just because one search pass finished.
Do not stop just because one draft exists.
Stop only when all of the following are true:
- every must-answer question is either answered or explicitly marked unresolved
- every load-bearing claim in the final deliverable has evidence
- no unresolved high-severity contradictions remain
- the verifier passes
- the deliverable satisfies the output contract
- the relevant quality reviewers pass or return only minor issues
- the evidence confidence for the main recommendations is acceptable for the task class
- either
  - the evidence coverage threshold is met
  - or another search round yields low marginal gain
  - or another draft round yields low marginal gain
Use a diminishing-returns rule, for example:
- if the last N searches add no new high-value sources or no new supported claims, the orchestrator may stop research
- if the last N review cycles produce only cosmetic edits, the orchestrator may stop drafting
Make this configurable.

Research-grade plus deliverable-grade loop:
Implement a controlled loop:
intake -> select method -> route experts -> plan -> search -> triage -> grade evidence -> design deliverable -> draft -> verify -> review -> search or redraft again if needed -> final
Add hard caps for:
- total iterations
- total source opens
- total time
- token budget if available
When a cap is hit, finalize honestly with a partial-completion report.

Guardrails:
- Do not fabricate citations.
- Do not cite a source that was not actually retrieved or recorded.
- Prefer official or primary sources for factual and technical claims.
- Surface disagreement instead of flattening it.
- If a claim is uncertain, label it uncertain.
- Do not expose private chain-of-thought. Visible reasoning should be a concise progress summary only.
- Keep irreversible actions out of scope unless explicitly enabled later.
- Make network access optional via config, but when enabled, internet search must work end to end.
- Never allow the synthesizer to introduce domain-specific claims that are not supported by the evidence ledger or explicitly marked as assumptions.
- Never let elegant writing hide weak evidence.
- Never let strong research excuse weak output structure.
- Never let a technically correct answer ignore the actual audience.
- Never let a recommendation appear without a clear basis, assumptions, and limitations.
- Never let spreadsheet outputs ship without checks, review notes, and auditability.
